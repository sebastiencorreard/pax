# 08 — Internationalisation

This document describes how OEF exercises declare and use a language, how multi-language titles work, how the runtime loads language-specific UI strings, and the character-encoding history.

---

## `\language{code}` Directive

The `\language` directive declares the primary language of the exercise.

Handler: `p_language()` in `WIMS/src/OefMsg2wims/process.c:101-106`:

```c
void p_language(char *p[MAX_PARM])
{
    p[0]=find_word_start(p[0]);
    *find_word_end(p[0])=0;
    if(strlen(p[0])==2) fprintf(outf,"language=%s\n",p[0]);
}
```

Key constraint: **the language code is only stored when it is exactly 2 characters long.** Longer strings (e.g. full language names like `"french"` or `"dutch"`) are silently ignored.

This is why PAX's import script must normalise full language names to ISO 639-1 codes before inserting exercises into the database (noted in the PAX `CLAUDE.md`).

### Accepted codes

Any two-character string is accepted; the common values are:

| Code | Language |
|---|---|
| `fr` | French |
| `en` | English |
| `nl` | Dutch |
| `de` | German |
| `es` | Spanish |
| `it` | Italian |
| `ca` | Catalan |
| `cn` | Chinese |
| `si` | Slovenian |

These correspond to the language-specific subdirectories under `WIMS/public_html/scripts/oef/` (e.g. `oef/fr/`, `oef/en/`, `oef/nl/`).

## Multi-Language Titles

The directives `\title_fr{}`, `\title_en{}`, `\title_nl{}`, etc. are listed as separate entries in `directives[]` with type `t_lang` (`WIMS/src/OefMsg2wims/oef2wims.c:105-112`):

```c
{"title_ca", t_lang, 1, p_header},
{"title_cn", t_lang, 1, p_header},
{"title_en", t_lang, 1, p_header},
{"title_es", t_lang, 1, p_header},
{"title_fr", t_lang, 1, p_header},
{"title_it", t_lang, 1, p_header},
{"title_nl", t_lang, 1, p_header},
{"title_si", t_lang, 1, p_header},
```

For type `t_lang`, the second argument `parm[1]` is set to the directive name itself (e.g. `"title_fr"`) by `process()` (`WIMS/src/OefMsg2wims/oef2wims.c:226-228`):

```c
case t_head2: case t_lang: {
    define[define_no].parm[1]=directives[i].name;
    break;
}
```

`p_header()` then emits `title_fr=<text>`, `title_en=<text>`, etc. in the `.def` header.

At runtime, `Main.phtml` selects the title for the current language:
```
!if $(title_$lang)!=$empty
  !set title=$(title_$lang)
!endif
!if $(title_$lang_choice)!=$empty
  !set title=$(title_$lang_choice)
!endif
```
(`WIMS/public_html/scripts/oef/Main.phtml:25-30`)

`$lang_choice` takes precedence over `$lang`, allowing the user's selected interface language to override the detected browser language.

## Language-Specific UI Strings

The runtime loads language-specific strings from `oef/$lang/names.proc` (`WIMS/public_html/scripts/oef/var.proc:25-29`):

```
!read oef/$lang/names.proc
!if $lang_exists!=yes
  !let modu_lang=en
  !read oef/$modu_lang/names.proc
!endif
```

If the language is not supported (`lang_exists` is not set to `yes` by the names file), the runtime falls back to English.

`names.proc` defines all UI strings as WIMS variables such as:
- `name_enterreply` — "Enter your reply"
- `name_sendanswer` — "Submit answer(s)"
- `name_Solution` — "Solution"
- `name_empty` — error message for empty fields
- `ch_choose`, `ch_noidea`, `ch_none`, `ch_good1`, `ch_goodn` — choice labels

Example from `WIMS/public_html/scripts/oef/en/names.proc`:
```
!distribute item choose, I have no idea, none of the above is correct,\
    the correct answer is, the correct answers are\
    into ch_choose, ch_noidea, ch_none, ch_good1, ch_goodn
```

These strings appear in the form rendering and answer analysis templates and are fully language-specific.

## `\choice` "I Don't Know" Localisation

The "I don't know" label in multiple-choice questions comes from `ch_noidea`, which is set by `names.proc` in each language. The `\choice` directive itself has no language-specific processing in the compiler — all localisation happens at runtime through the language-specific name variables.

The `noidontknow` option in `choiceoption<n>` suppresses the "I don't know" item from the rendered form (handled in `anstype/menu.input` and related form scripts).

## Language in `oef_lang` and Translation

`var.proc` sets:
```
oefenv_lang=$lang
!if $translation_language!=$empty and $lang_choice!=$empty
  oefenv_lang=$lang_choice
!endif
```
(`WIMS/public_html/scripts/oef/var.proc:33-38`)

`$oefenv_lang` is used internally for language-dependent answer processing (e.g., `atext` uses `scripts/oef/$module_language/atext.dic` for suffix rules).

The `translation_language` variable (set from the module's definition) enables a distinction between the exercise's native language and the student's interface language.

## Character Encoding History

- **Historical**: OEF files were written in ISO-8859-1 (Latin-1). This was common for Western European languages in the 1990s–2000s when WIMS was created.
- **Current**: New exercises use UTF-8.
- **Compiler handling**: The `OefMsg2wims` compiler is encoding-agnostic — it operates byte-by-byte and does not interpret multi-byte sequences. Any valid UTF-8 or ISO-8859-1 file will be processed correctly as long as no byte in the range 0–6 appears (those are used as sentinels, and the sanitisation pass in `getinp()` replaces bytes 0–6 with spaces; `WIMS/src/OefMsg2wims/oef2wims.c:160-162`).
- **Runtime**: The WIMS HTTP server sets the content-type charset. Modern deployments use UTF-8 throughout.
